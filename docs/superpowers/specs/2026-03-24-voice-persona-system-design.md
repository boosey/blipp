# Voice Persona System — Cross-Provider Consistency

## Problem

When the TTS model chain falls back from the primary provider to a secondary or tertiary provider, users hear a different voice. The current voice preset system stores per-provider config but does not enforce completeness or signal when a fallback voice was used. This creates an inconsistent listening experience.

## Solution

Rework voice presets into abstract voice personas with:

1. **Curated cross-provider mappings** — each persona has hand-picked best-match voices for all active TTS providers
2. **Fallback degradation flagging** — clips generated by non-primary providers are flagged, and the frontend shows a subtle indicator
3. **Improved admin tooling** — side-by-side provider editor with audio preview and completeness validation
4. **Simplified user UX** — users see persona names and descriptions only, no provider details

## Design

### Data Model

Add to `VoicePreset`:
- `voiceCharacteristics Json?` — `{ gender: "female"|"male"|"neutral", tone: "warm"|"calm"|"energetic"|"neutral", pace: "steady"|"fast"|"slow" }`. Descriptive metadata now; becomes the auto-matching taxonomy later.

Add to `Clip`:
- `voiceDegraded Boolean @default(false)` — set when audio was generated by a non-primary TTS provider during fallback.

The existing `VoicePreset.config` JSON shape is unchanged: `{ openai: { voice, instructions, speed }, groq: { voice }, cloudflare: {} }`. The change is enforcement: all providers must be populated for active personas.

### Fallback Behavior

In `worker/queues/audio-generation.ts`, after the model chain loop succeeds:

- If the successful attempt was not the primary (index > 0), set `voiceDegraded: true` on the Clip
- Log a structured pipeline event with `{ voiceDegraded: true, primaryProvider, actualProvider }`
- No behavior change to the fallback itself — audio is still delivered

### Audio Preview API

Two new endpoints for generating short TTS samples:

**Admin:** `POST /api/admin/voice-presets/preview`
- Body: `{ provider: string, voice: string, instructions?: string, speed?: number, text?: string }`
- Returns: audio/mpeg stream
- Uses the specified provider's TTS directly

**Public:** `POST /api/voice-presets/:id/preview`
- No body (uses primary provider + preset config)
- Returns: audio/mpeg stream
- Gated by plan access via `checkVoicePresetAccess()`

Both use a short default text: "Here's a quick preview of how this voice sounds for your daily briefing."

### Admin Persona Editor

The existing admin voice presets page is reworked:

- Three-column layout showing OpenAI, Groq, and Cloudflare config side by side
- Voice dropdown per provider with all available voice options
- Preview button per provider column — plays a short TTS sample inline
- Voice characteristics section: gender, tone, pace dropdowns
- Save validation: all active providers must have voice mappings

### User Voice Picker

Simplified to abstract personas:

- Shows persona name and one-line description
- Preview button generates a sample using the primary TTS provider
- Locked personas show plan upgrade prompt
- No provider names, model IDs, or technical details visible

### Frontend Degradation Indicator

When a briefing's clip has `voiceDegraded: true`:
- Show a small info icon next to the briefing in the feed
- Tooltip: "This briefing used an alternate voice due to a temporary service issue"
- Non-intrusive — informational only

### Initial Personas

Seed 3-5 system personas with complete cross-provider mappings:

| Persona | Description | OpenAI | Groq | Characteristics |
|---------|------------|--------|------|-----------------|
| Nova | Warm, professional | coral | tara | female, warm, steady |
| Echo | Calm, measured | sage | austin | male, calm, measured |
| Spark | Energetic, upbeat | shimmer | kai | female, energetic, fast |

Exact secondary/tertiary provider mappings will be refined through listening tests during implementation. Free plan gets Nova + Echo; paid plans unlock all.

## Out of Scope

- **Background re-generation** of degraded clips when primary provider recovers
- **Automatic voice matching** algorithm based on characteristics taxonomy
- **Cloudflare Workers AI** voice matching (limited voice catalog — may not be viable)

## Future: Auto-Matching

The `voiceCharacteristics` field is laid as groundwork. When the voice catalog grows beyond ~10 personas, build a matching function that:
1. Takes a persona's characteristics
2. Queries each provider's voice catalog metadata
3. Returns the best match per provider

This replaces the manual mapping for non-curated voices while curated personas keep their hand-picked mappings.
