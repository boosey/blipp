# Briefing Audio Assembly & Processing

**Date:** 2026-03-14
**Status:** Draft — awaiting review

## Problem

Blipp briefings are raw TTS MP3 files with no structure. There's no intro/outro, no sonic branding, no metadata spoken at the start. When a user hits "play all" on their feed, clips run together with nothing to mark where one ends and the next begins.

Stage 5 (Briefing Assembly) is data-only — it links Clip records to Briefings but produces no audio output.

## Goal

Each briefing should be a complete, polished audio package:

```
[intro jingle] + [clip MP3] + [outro jingle]
```

The narrative text (generated by the LLM in Stage 3) should open with spoken metadata:

> "From **The Daily**, episode **The Election Results**, released March 12th. Originally 45 minutes — here's your 5-minute briefing."

When users play feed items back-to-back, the intro/outro jingles naturally delineate boundaries.

## Phased Approach

### Phase 1: Frame-Level MP3 Assembly (No Wasm)

**No decoding. No encoding. No PCM. Just binary MP3 concatenation.**

MP3 frames are independently decodable — you can concatenate MP3 files at the binary level and any player will play them seamlessly. Blipp already has `concatMp3Buffers()` in `worker/lib/mp3-concat.ts` that does exactly this.

**What Phase 1 delivers:**

1. **Intro jingle** — Pre-made MP3 asset prepended to every briefing
2. **Outro jingle** — Pre-made MP3 asset appended to every briefing
3. **Narrative metadata intro** — LLM prompt updated to include spoken podcast name, episode title, release date, original length, and clip length at the start of every narrative
4. **Assembled output** — Single MP3 file stored in R2: `[intro.mp3] + [clip.mp3] + [outro.mp3]`

**Memory usage:** Trivial. A 30-min clip at 128kbps = ~28MB. Intro + outro jingles = ~200KB each. Total: ~29MB. Well under 128MB.

**CPU usage:** Near zero. Binary array concatenation.

**What it does NOT do:**
- No volume normalization (intro jingle must be pre-mastered to match TTS output levels)
- No crossfading (hard cuts between jingle and clip — acceptable if jingles have built-in fades)
- No mixing (no background music under speech)

### Phase 2: PCM Processing via Wasm

**Decode MP3 → manipulate PCM samples → re-encode to MP3.**

This is where Wasm enters. Only needed when you want to actually modify the audio signal.

**What Phase 2 adds:**

1. **Volume normalization** — Ensure consistent loudness across clips from different TTS providers/voices
2. **Crossfading** — Smooth transitions between jingle and clip instead of hard cuts
3. **Music bed mixing** — Low-volume background music under speech
4. **Ad audio insertion** — Mix/overlay ad audio (uses existing `adAudioUrl`/`adAudioKey` schema fields)

**Library approach (npm-first):**
- **Encoding:** `wasm-media-encoders` — LAME compiled to Wasm (66KB gzipped), streaming API
- **Decoding:** TBD — test `mpg123-decoder` or similar in Workers runtime
- **Processing:** TypeScript math on Float32Arrays

**Graduation path:** If npm packages don't work in Workers runtime or performance is insufficient, build a custom Rust Wasm module (`symphonia` for decode + vendored LAME for encode).

## Phase 1 Technical Design

### Changes by Stage

#### Stage 3: Narrative Generation (Prompt Change)

Update the narrative generation prompt to instruct the LLM to include a spoken intro with episode metadata. The narrative stage already has access to all required data:

- `episode.title` — Episode title
- `episode.podcast.title` — Podcast title
- `episode.publishedAt` — Release date
- `episode.durationSeconds` — Original episode length
- `clip.durationTier` — Target clip length in minutes

**Prompt addition:**
```
Begin the narrative with a brief introduction stating:
- The podcast name
- The episode title
- When it was released (relative date like "released yesterday" or "from March 12th")
- The original episode length
- The briefing length

Example: "From The Daily, episode The Election Results, released yesterday.
Originally 45 minutes — here's your 5-minute briefing."

Then proceed directly into the content.
```

This requires no code changes to the audio pipeline — the metadata intro is part of the narrative text that goes through TTS as normal.

#### Stage 4: Audio Generation (No Change)

TTS generates MP3 from narrative text. The narrative now includes the spoken metadata intro, but nothing changes in the audio generation code.

#### Stage 5: Briefing Assembly (Audio Assembly)

Currently data-only. Add MP3 concatenation:

```typescript
// After creating Briefing records and before marking FeedItems READY:

// Load the clip's audio from R2
const clipAudio = await getWorkProduct(env.R2, wpKey({
  type: "AUDIO_CLIP",
  episodeId: job.episodeId,
  durationTier: job.durationTier,
  voice: "default",
}));

if (clipAudio) {
  // Load jingle assets from R2
  const intro = await getWorkProduct(env.R2, "assets/intro.mp3");
  const outro = await getWorkProduct(env.R2, "assets/outro.mp3");

  // Assemble: [intro] + [clip] + [outro]
  const parts: ArrayBuffer[] = [];
  if (intro) parts.push(intro);
  parts.push(clipAudio);
  if (outro) parts.push(outro);

  const assembled = concatMp3Buffers(parts);

  // Store assembled briefing in R2
  const briefingR2Key = wpKey({
    type: "BRIEFING_AUDIO",
    userId: fi.userId,
    date: new Date().toISOString().slice(0, 10),
  });
  await putWorkProduct(env.R2, briefingR2Key, assembled);

  // Update Briefing with audio key
  await prisma.briefing.update({
    where: { id: briefing.id },
    data: { adAudioKey: briefingR2Key }, // or a new field: assembledAudioKey
  });
}
```

#### Serving

The clip endpoint (`GET /api/clips/:episodeId/:durationTier.mp3`) serves raw clips. Assembled briefings need a separate endpoint or the existing endpoint needs to serve the assembled version:

```
GET /api/briefings/:briefingId/audio → assembled MP3 from R2 (intro + clip + outro)
GET /api/clips/:episodeId/:durationTier.mp3 → raw clip (no jingles, for admin/debug)
```

### New Files

```
worker/lib/audio/
  assembly.ts       — assembleBriefing(): concat intro + clip + outro
  types.ts          — BriefingAudioConfig type
```

Reuses existing `worker/lib/mp3-concat.ts` for the actual concatenation.

### R2 Asset Storage

```
assets/
  intro.mp3         — Briefing intro jingle (pre-mastered to match TTS levels)
  outro.mp3         — Briefing outro jingle (pre-mastered)
```

Uploaded via admin UI or CLI (`wrangler r2 object put`). These should be:
- Short (2-5 seconds each)
- Pre-mastered to match expected TTS output volume (~-16 LUFS for speech)
- Have built-in fades (fade-in on intro, fade-out on outro) since Phase 1 does hard cuts
- MP3 format, same sample rate as TTS output (24kHz mono recommended)

**R2 metadata:** All audio stored via `putWorkProduct` should include `httpMetadata: { contentType: "audio/mpeg" }`.

### Schema Consideration

The `Briefing` model currently has `adAudioUrl` and `adAudioKey` but no field for the assembled audio. Options:

1. **Reuse `adAudioKey`** — semantically wrong but works
2. **Add `assembledAudioKey`** — clean but requires migration
3. **Derive from convention** — the assembled audio is always at the BRIEFING_AUDIO WorkProduct key, look it up

**Recommended:** Option 3 for Phase 1. The WorkProduct with `type: BRIEFING_AUDIO` already has an R2 key. The serving endpoint looks up the WorkProduct. No schema change needed.

## Error Handling

Audio assembly must never block briefing delivery:

```typescript
async function assembleBriefingAudio(
  clipAudio: ArrayBuffer,
  env: Env,
  log: Logger
): Promise<ArrayBuffer> {
  try {
    const intro = await getWorkProduct(env.R2, "assets/intro.mp3");
    const outro = await getWorkProduct(env.R2, "assets/outro.mp3");

    const parts: ArrayBuffer[] = [];
    if (intro) parts.push(intro);
    parts.push(clipAudio);
    if (outro) parts.push(outro);

    return concatMp3Buffers(parts);
  } catch (err) {
    log.warn("assembly_fallback", { error: err instanceof Error ? err.message : String(err) });
    return clipAudio; // Fall back to raw clip
  }
}
```

If jingle assets aren't uploaded yet, the briefing is just the raw clip. No errors, no blocking.

### Feature Toggle

```typescript
const assemblyEnabled = await getConfigValue(prisma, "BRIEFING_ASSEMBLY_ENABLED", "false");
```

Consistent with existing `checkStageEnabled` pattern. Allows enabling assembly after jingle assets are uploaded and tested.

## Memory Budget

Phase 1 operates entirely on compressed MP3 data. No PCM decode.

| Clip Duration | MP3 Size (128kbps) | + Jingles | Total |
|---|---|---|---|
| 1 min | ~1 MB | ~0.4 MB | ~1.4 MB |
| 5 min | ~5 MB | ~0.4 MB | ~5.4 MB |
| 15 min | ~14 MB | ~0.4 MB | ~14.4 MB |
| 30 min | ~28 MB | ~0.4 MB | ~28.4 MB |

**Peak memory: ~29MB** for the largest tier. Well under 128MB. No streaming needed in Phase 1.

Phase 2 (PCM processing) would use chunked streaming for clips > 10 minutes. See the Phase 2 section above for the streaming strategy.

## Constraints & Risks

### Phase 1

- **Zero risk from Wasm compatibility** — Phase 1 uses no Wasm
- **Zero risk from memory** — compressed MP3 only, max ~29MB
- **Zero risk from CPU** — binary concatenation is near-instant
- **Jingle quality matters** — must be pre-mastered to match TTS volume and sample rate. If the intro is at a different sample rate than the clip, some players may glitch at the boundary. Mitigate by encoding jingles at 24kHz mono to match TTS output.
- **Hard cuts** — no crossfading between jingle and clip. Mitigate by building fades into the jingle audio files themselves.

### Phase 2

- **Wasm runtime compatibility** — `wasm-media-encoders` was built for browser/Node. Needs spike test.
- **MP3 decoder availability** — No confirmed Workers-compatible streaming decoder. Fallback: compile `puremp3` Rust crate.
- **Sample rate mismatches** — Different TTS providers output at different rates. Process at native rate; resample only when mixing.
